# info-proxy 設計書

## 1. 概要

`info-proxy` は、特定の Web サイトに対してログインし、CSV ファイルなどの出力データを取得して内部に保存するための Go 製プログラムである。

利用者はまず対象サイトを選択し、その後は `info-proxy` がリバースプロキシとして動作して対象サイトのログイン画面を表示する。ログイン後は、対象サイトの CSV 出力ボタンを自動で押下し、出力された CSV を `info-proxy` 内部に保存する。

## 2. 目的

- サイトごとのログインと CSV 取得手順を共通化する
- 利用者が対象サイトへ直接アクセスしなくても処理を進められるようにする
- 取得した CSV を `info-proxy` 内部で管理し、後続処理につなげやすくする
- 取得処理の記録と障害調査をしやすくする

## 3. 想定ユースケース

- 社内システムから特定サイトの CSV を定期取得する
- 人手によるログインと CSV ダウンロード作業を省力化する
- 複数サイトの CSV 取得を同じ仕組みで扱う

## 4. 利用フロー

1. 利用者が `info-proxy` 上で対象サイトのボタンを選択する
2. `info-proxy` が対象サイトのログイン画面を表示する
3. 利用者がそのまま対象サイトへログインする
4. ログイン完了後、`info-proxy` が対象サイトの CSV 出力ボタンを自動で押下する
5. 出力された CSV ファイルを `info-proxy` 内部に保存する
6. 必要に応じて保存済み CSV を参照またはダウンロードできるようにする

## 5. 要件

### 5.1 機能要件

- 複数サイトの中から対象サイトを選択できる
- 選択したサイトのログイン画面へ誘導できる
- リバースプロキシとして対象サイトの画面を表示できる
- ログイン後の遷移を維持できる
- 対象サイトの CSV 出力ボタンを自動で押下できる
- 出力された CSV を内部ストレージに保存できる
- 保存済み CSV の一覧取得とダウンロードができる
- 実行ログを記録できる

### 5.2 非機能要件

- Go の標準的な開発・運用方法に沿う
- 設定は環境変数または設定ファイルから読み込める
- 失敗時に原因が追いやすいエラー情報を返す
- サイトごとの挙動差を吸収できるよう責務を分離する
- 保存した CSV の取り扱いを明確にし、誤上書きや取り違えを防ぐ

## 6. 全体構成

### 6.1 構成イメージ

- `main`
  - 起動処理
  - 設定読み込み
  - HTTP サーバ起動
- `config`
  - 環境変数や設定ファイルの読み込み
- `handler`
  - サイト選択やファイル操作の HTTP 受け口
- `proxy`
  - 対象サイトへのリクエスト中継
- `automation`
  - CSV 出力ボタンの自動押下などの操作処理
- `storage`
  - CSV の保存・一覧・取得
- `middleware`
  - ログ、認証、リクエスト ID 付与など
- `internal`
  - 共通ロジック

### 6.2 処理フロー

1. アプリケーション起動時に設定を読み込む
2. サイト定義と保存先を初期化する
3. 利用者のサイト選択を受け付ける
4. 対象サイトのログイン画面をリバースプロキシ経由で表示する
5. ログイン後の状態を維持する
6. CSV 出力可能な状態になったら自動操作を実行する
7. CSV を取得して内部ストレージへ保存する
8. 保存結果を返却し、ログへ記録する

## 7. 画面・操作設計

### 7.1 サイト選択画面

- 利用者が対象サイトを選ぶための入口画面を用意する
- サイトごとに識別子、表示名、接続先を持たせる

### 7.2 ログイン画面表示

- 選択されたサイトのログイン画面を、そのまま対象サイトの画面として見せる
- ブラウザから見た操作感を大きく変えない
- セッション情報や Cookie を維持する

### 7.3 CSV 取得操作

- ログイン後、対象サイトの CSV 出力ボタンを自動押下する
- 出力完了後、ダウンロードされた CSV を受け取る
- 取得失敗時は再試行や失敗理由の記録を行う

## 8. API 設計

### 8.1 エンドポイント案

- `GET /healthz`
  - 稼働確認
- `GET /sites`
  - 利用可能なサイト一覧
- `GET /site/{siteID}`
  - 対象サイトの入口
- `ANY /proxy/{siteID}/*`
  - 対象サイトへの中継
- `POST /site/{siteID}/csv`
  - CSV 出力処理の実行
- `GET /csv`
  - 保存済み CSV の一覧
- `GET /csv/{fileID}`
  - 保存済み CSV の取得

### 8.2 レスポンス方針

- 正常時は必要な情報を返す
- 異常時は共通フォーマットでエラーを返す
- CSV 取得成功時は保存先情報やファイル識別子を返す

## 9. 設定項目

### 9.1 例

- `PORT`
- `LOG_LEVEL`
- `SITE_CONFIG_PATH`
- `STORAGE_DIR`
- `REQUEST_TIMEOUT`
- `RETRY_COUNT`
- `ENABLE_REQUEST_LOG`

### 9.2 サイト定義例

- サイト識別子
- 表示名
- ベース URL
- ログイン開始 URL
- CSV 出力ボタンの特定方法
- 保存名のルール

### 9.3 設定管理方針

- 起動時に設定を一括ロードする
- 必須項目が欠けている場合は起動を失敗させる
- サイトごとの差分は設定で切り替える

## 10. 保存設計

- CSV は `info-proxy` 内部のストレージへ保存する
- 保存先はファイルシステムまたは将来の外部ストレージを想定する
- ファイル名にはサイト ID、取得日時、連番などを含める
- 同名衝突を避けるため一意性を持たせる
- 保存済みデータにはメタ情報を付与する

## 11. ログ設計

- 1 リクエストごとに request ID を付与する
- サイト選択、ログイン、CSV 取得、保存の各イベントを記録する
- 取得失敗時は対象サイト、処理段階、失敗理由を残す
- 個人情報や秘密情報はマスクする

## 12. エラーハンドリング

- 入力不正は 4xx を返す
- 対象サイト側の障害は 5xx または再試行対象として扱う
- 内部例外は詳細をログに残し、外部には最小限の情報のみ返す

## 15. 実装時の注意

- handler、proxy、automation、storage を分離する
- 設定値をコードに直書きしない
- テストしやすいように依存を注入する
- ブラウザ操作や自動押下の失敗を前提にリトライと停止条件を設ける
- 保存対象の CSV が正しいサイト・正しい日時のものか確認できるようにする
